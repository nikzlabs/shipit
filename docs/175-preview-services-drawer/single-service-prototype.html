Single-service drawer — thinking past the floating chip

Reframe: one service isn't “a list of one”, it's the focus. With many services density wins; with one, the drawer is all yours — so fill it with something useful instead of stranding a tiny card.

The most useful thing to fill it with is the service's own live log — which is exactly what this drawer was built to show (docs/175). Header stays full-width, so the collapsed state never changes.

DECIDED & SHIPPED → Direction #2 (Focus card) for a running/starting single service, with the log shown directly (no “Open full log” button), plus #3's Status hero for the stopped state (a blank terminal would be a poor resting state). And a global fix: all controls grouped on the left so they aren't stranded at the far edge on wide monitors. The directions below are the record of how we got here.

1

Console — the single service is its log

my pick

No list, no card. The header morphs into the service's toolbar (it already does this when a service is selected) and the body becomes the live terminal, full-bleed. Zero void, maximally useful, and it reuses machinery that already exists. Two sub-states below: booting, then running.

Focus card — full-width card, log shown directly

chosen

Final design. A full-width card: identity + controls grouped on the left (no far-edge travel), and the live log right there beneath — no “Open full log” hop, it's already the log. Note the action buttons sit next to the name, not pushed to the right edge; the wide log below justifies the card's width so the top row's right-side gap reads fine.

3

Status hero — centered, for a stopped / crashed single service

When the one service is stopped or crashed there's no log worth showing, so don't fake a console. Treat it like an intentional empty-state: centered icon, name, status, one big primary action. Centered reads as deliberate here (unlike centering a normal card) because it's a true focal moment. Flips to #1 the instant it boots.
