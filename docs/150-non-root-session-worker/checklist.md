# Non-root Session Worker Checklist

## Implemented in this PR (SHI-31) — flag-off-by-default

The whole migration is gated on `SHIPIT_SESSION_WORKER_UID`. Unset (default) =
byte-for-byte today's behavior: the entrypoint execs the worker as **root** with
no chown, and the orchestrator's chown helpers are no-ops. Set (e.g. `1000`) =
the entrypoint chowns the mounts + drops to `shipit`, and the orchestrator hands
ownership of its post-boot writes to the same UID. Nothing here changes prod
behavior until the var is set.

- [x] Add a stable `shipit` UID/GID to session worker images (rename upstream `node` → `shipit`, UID/GID 1000).
- [x] Root entrypoint that prepares writable mounts and drops to `shipit` — **gated** on `SHIPIT_SESSION_WORKER_UID` (`docker/session-worker/entrypoint.sh`, `gosu`). Unset → exec as root; no privilege drop, no chown.
- [x] Shared `agentHome()` runtime resolver (`src/server/shared/agent-home.ts`), resolved at call time. Default `/home/shipit`; local mode pins `/root`.
- [x] Replace `/root` assumptions in worker, Claude, Codex, terminal, and the container env builder.
- [x] Move credential symlinks to `/home/shipit`; npm global prefix → `~/.npm-global` (§6); Playwright browsers pinned to `/opt/playwright-browsers` (§8).
- [x] Orchestrator-side ownership handoff gated on `SHIPIT_SESSION_WORKER_UID` (§7): per-session credentials (incl. `provisionRepoMemory` so auto-memory stays writable), container gitconfig, user uploads, CI-fix logs.
- [x] Keep local/dogfood mode on root with `AGENT_HOME=/root` (§9).
- [x] Ops `.docker` image: add `shipit` to `systemd-journal`/`adm` so host journal reads survive. **Fixed (post-deploy):** the original `usermod -aG systemd-journal,adm shipit || true` shipped `shipit` in *no* supplementary groups (`groups=1000(shipit)`), killing the journal pillar in every ops session. `usermod -aG a,b` is atomic — the `systemd` install doesn't reliably create the `systemd-journal` group at build time, so the whole add failed and `|| true` masked it. Now `groupadd -rf` both groups first, add membership per-group, and assert the result so a regression fails the build instead of silently disabling journal reads.
- [x] Enumerate the writable paths the worker needs (groundwork for SHI-97 ReadonlyRootfs/seccomp) — see `plan.md` "Writable paths".
- [x] Hide the `.shipit-uid-1000` chown sentinel from the file tree/watcher.
- [x] Update `src/server/shipit-docs/environment.md`.
- [x] Startup fail-fast guard on `SHIPIT_SESSION_WORKER_UID` drift (`worker-uid-guard.ts`, wired into `buildApp` for containerized prod). Persists a per-boot marker; refuses to start on a non-root → unset rollback with sessions present, unless `SHIPIT_SESSION_WORKER_UID_ALLOW_DOWNGRADE=1`.
- [x] Unit/integration tests: `agent-home`, `session-worker-uid`, `worker-uid-guard`, `container-lifecycle` env, `session-credentials` §7 chown (creds + memory).
- [x] Run lint, typecheck, and targeted container/worker tests.

## Nothing reverted

Reviewed for parts to back out: none. The earlier asymmetry (image dropped to
`shipit` unconditionally while the orchestrator chowns were gated, which would
break auth if the new image shipped with the var unset) was fixed by gating the
entrypoint's privilege drop on the same var — not by reverting any feature.

## Activation-blocker fix — deploy validation found two gaps (SHI-31)

A real deploy validation with `SHIPIT_SESSION_WORKER_UID=1000` surfaced two
gaps the original §7 reasoning missed. Both are the **gate to flipping the var
in prod**; fixed in this PR, still flag-gated.

- [x] **Root orchestrator git refused on worker-owned worktrees (`detected dubious ownership`).** §7 reasoned only about *write* ownership and never anticipated that root then can't *read/operate* git over a 1000-owned tree. Broke auto-commit, auto-push, branch graduation, the bare-cache fetch, and the `gh pr` branch lookup. Fixed by adding `safe.directory=*` to the orchestrator's **global** git config in `initGlobalGitConfig` — gated on `sessionWorkerUid()`. `safe.directory` is honored only from system/global config (never `-c` / repo-local), and the orchestrator's `GIT_CONFIG_GLOBAL` is root-owned so git-as-root trusts it; one `*` covers the bare cache + every per-session worktree + the `gh`/PR git path (all inherit `GIT_CONFIG_GLOBAL`). Running git as uid 1000 was rejected — it would break on the root-owned bare cache and the root-owned global config.
- [x] **Post-boot orchestrator git/compose writers left files root:root.** Finished the §7 chown wiring for the writers that were missed: `compose-generator.writeComposeOverride` (the override file **and** its `.shipit` dir — this also clears the transient `EACCES /workspace/.shipit/.install-done` during warm-claim, since the dir is now worker-owned), the post-turn auto-commit (`ws-handlers/post-turn.ts` hands `.git` back on every path), `repo-git.cloneFromCache` (post-boot reclone), and the claim / warm-pool / fork-merge / child-session git ops. New helper `chownWorkspaceGitToSessionWorker(workspaceDir)` chowns `<dir>/.git`.
- [x] Tests: `safe.directory` gated write (git-config), `chownWorkspaceGitToSessionWorker` (session-worker-uid), compose-override chown (compose-generator).
- [x] **Overlay dep store `node_modules` was root-owned — agent couldn't `npm install` (SHI-145).** The docs/183 overlay dep store (default-on, incl. prod) mounts each declared dep dir as a separate `type=overlay` volume at `/workspace/<dep-dir>`, whose per-session upper/work dirs and shared base are created by the **root** orchestrator and were never handed to the worker uid — so the non-root worker (1000) EACCESed on `npm install` into the overlay, while the rest of `/workspace` was 1000. The entrypoint's one-shot `/workspace` `chown -R` doesn't reliably cover the separately-mounted overlay, hence the *inconsistent* ownership across sessions on the same scope. Fixed symmetric to §7, gated on `sessionWorkerUid()`: **(a)** `prepareOverlayDirs` (`container-lifecycle.ts`) chowns each per-session **upper/work** dir to the worker uid right after mkdir — so a NEW dep write lands in the upper as the worker (overlayfs stamps the new upper file with the writer's fsuid, which requires a worker-writable upperdir). **(b)** `publishBase`→`finalize` (`overlay-base.ts`) chowns the freshly-materialized **base generation** to the worker uid before the pointer is written (race-free: no session has mounted it yet) — so overlayfs copy-up of an EXISTING base dep, which preserves the *lower* file's ownership, produces a worker-owned writable upper file instead of a root-owned EACCES. Hardlink-dedup shares inodes with the prior generation, but that generation is itself worker-owned, so the recursive chown only flips the newly-copied files. The empty cold-start `g0` lowerdir is left root-owned-but-traversable (cold-start writes all go to the upper). Both no-op when the flag is unset. Tests: `overlay-base` chown-spy (created + advanced, not on skip), `container-lifecycle` `prepareOverlayDirs` (mkdir + worker-uid handoff of upper/work). **Verified live** (dev, flag on, orchestrator root): `prepareOverlayDirs` chowns upper/work → `1000:1000` while the cold-start `g0` lowerdir stays root-traversable.
- [x] **Fork-merge left the merged worktree root-owned (SHI-144 analog).** Same bug class, found during SHI-31 dev validation: `mergeSession` (`services/session-fork-merge.ts`) runs `git.merge` as the root orchestrator against the active session's *booted* clone — rewriting worktree files — but its `finally` handed only `.git` back (line 192), so the merged worktree files stayed root-owned and the non-root agent couldn't edit them on its next turn. Fixed by also calling `chownWorktreeToSessionWorker(activeSessionDir, depDirs)` (dep-dir-aware, bounded) alongside the `.git` chown. New `session-fork-merge.test.ts` asserts both handoffs fire after a clean merge and on the merge-throws path (finally). Gated/no-op when the flag is unset.
- [x] **Playwright MCP browser broken under non-root — `EACCES mkdir` into the read-only browser store (SHI-145 regression).** A live session hit `EACCES: permission denied, mkdir '/opt/playwright-browsers/mcp-chrome-for-testing-<hash>'` on every `browser_*` call, blocking browser tools + `present`-tab screenshot verification. **Not** a browser-download mismatch (the initial hypothesis): the `chromium-<rev>` binary (= Chrome for Testing) is correctly pre-installed read-only per §8 and there is zero runtime download. Root cause: `@playwright/mcp` defaults to a **persistent** browser context and `mkdir`s its per-cwd *profile* dir (`mcp-<channel>-<sha256(cwd)[0:7]>`) under playwright-core's `registryDirectory`, which resolves to `PLAYWRIGHT_BROWSERS_PATH=/opt/playwright-browsers` — the §8 read-only root-owned store. As uid 1000 the profile `mkdir` EACCESes; before SHI-145 the root worker's `mkdir` into the root store silently succeeded. Fix: pass `--isolated` in the launch command (`src/server/session/agents/playwright-mcp.ts`, shared by the Claude + Codex adapters) so the profile lands in an ephemeral `os.tmpdir()` (`/tmp`, writable) and the §8 store stays read-only and shared. Reproduced the EACCES with the old args and confirmed `--isolated` makes `browser_navigate` return a snapshot, both as uid 1000 against the pinned `@playwright/mcp` 0.0.75. Regression guard: `playwright-mcp.test.ts` asserts `--isolated` stays in the launch command. See plan.md §8.1.
- [x] **Rebase driver left `.git` AND the worktree root-owned (SHI-144).** The user-driven rebase + auto-conflict-resolve flow (`services/rebase-driver.ts`) runs its git ops (fetch / rebase / `--continue` / stageAll / forcePush / abort) as the root orchestrator but dispatches resolution turns with `postTurn: "none"`, which elides the `postTurnCommit` handoff every normal turn gets. The root rebase re-roots **both** `.git` (so the non-root agent's in-container `git` fails) **and the worktree files** — including the conflicted files the agent must *edit* to resolve (so the resolution turn EACCESes even with `.git` handed back; caught by a prod-machine agent on the live conflict path before merge). Fixed by `handWorkspaceBackToWorker(runner.sessionDir)`, which hands back `.git` (object-aware `chownWorkspaceGitToSessionWorker`) **and** the worktree (`chownWorktreeToSessionWorker`, new helper — walks the worktree, skips `.git` + the declared `agent.dep-dirs` so it stays bounded by the source tree, not `node_modules`). Called (a) before each resolution turn (so the agent can both run git and edit conflicts), (b) in a `finally` wrapping `runRebaseFlow` (every exit: clean / resolved / up-to-date / abort / throw — also heals pre-existing root-owned worktree files), and (c) on the auto-path stale-rebase pre-flight abort + post-`Promise.race` (timeout-teardown abort). Gated/no-op when the flag is unset. Tests spy both handoffs (incl. asserting the worktree is handed back *before* the agent edits) across up-to-date / clean / conflict / throw; the real uid-1000 EACCES proof is the live dev validation.

- [x] **Recreate-after-idle left a stray root-owned worktree unrepaired — wedged the resume (SHI — "container wedged, root-owned workspace").** Found live on a long-lived parent session that fanned out 20+ children and was idle-disposed mid-fan-out: on recreate (to drain a queued merge-watch wake-turn) its `npm install` failed fast (~2.6 s) and the agent couldn't edit files or rebase. Root cause is the same §7 bug class at a NEW seam: the §7 handbacks all fire at git-op seams (claim/rebase/fork-merge/post-turn), but **container-recreate-after-idle runs none of them** — it boots a fresh container against the persisted clone and trusts it's already worker-owned, and the entrypoint's boot-time `chown -R /workspace` is skipped on every reboot (the UID sentinel already exists). So any root-owned node that slipped in during the session's life persists and is never repaired; the non-root agent (1000) then EACCESes — `npm install` fails fast writing into the root-owned cwd (the `install_ok=false` is a *downstream symptom*, not the cause → hypothesis **B**). Fixed by `selfHealWorkspaceOwnership(config, workspaceVolume)` → `handWorkspaceBackToWorker` in `createContainer` (`container-lifecycle.ts`), the single boot choke point — bounded (`.git`-fanout + dep-dir skip) and idempotent, so a recreated session is always left writable. Gated twice (flag unset → no-op; dev bind-mount → skip, never chown host source). Also chowned the orchestrator-root `preStampInstallMarker` `.shipit`+marker (`overlay-session.ts`) so the worker can later overwrite an invalidated marker. **Observability:** worker now folds a bounded stderr tail into the `install_error` message (`install-failure.ts`) and the orchestrator `console.error`s the failure (no-viewer recreate installs were emit-only → swallowed). Tests: `selfHealWorkspaceOwnership` gating (container-lifecycle), pre-stamp chown handoff (overlay-session), `formatInstallFailureMessage` (install-failure). See plan.md §7.2.
