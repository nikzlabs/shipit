Composer quote rendering

Three ways the same quoted action-card snapshot could appear in the message composer. Recommendation on the left; the two rejected alternatives on the right. Dark mode, ShipIt tokens.

✓ Recommended

Plain editable text + decoration-only left rail

Quote stays editable markdown >  lines — the textarea is the single source of truth. A backdrop accent (behind the text) just makes it read as a quote. Send-fidelity, voice, drafts, multi-quote all unchanged.

> Resolve action checklist — 3 items > [x] Add a history round-trip test > [ ] Wire toRow / fromRow > [ ] Register in CARD_MESSAGE_FIELDS Skip the third one for now, do the first two.
Opus 4.8

Why it wins: the user can still trim/annotate any [x] line, dictation appends into the same box, and the agent receives exactly the > text shown. The rail is pure decoration — delete it and nothing breaks.

✕ Rejected — rich editor

Styled blockquote rendered in-place

Prettiest, but needs contentEditable / a rich editor. That rewrites the composer core and re-implements IME, paste, @// autocomplete, voice-splice, draft serialization & a11y — for a quote bar.

Skip the third one for now, do the first two.

Cost vs. benefit upside-down: six working integrations lean on "value is a plain string with a cursor." A document model puts all of them at risk to make a quote look nicer.

✕ Rejected as default — reply chip

Removable context chip above the textarea

Good for reply-to flows, but the quote lives outside the editable text — so the user can't trim individual [x] lines, which is exactly what the action card needs. Adds a second source of truth.

Breaks editability: a non-editable chip can't be trimmed or annotated inline. Fine to revisit later, scoped only to pure reply-reference flows (chat quote-reply, doc replies) — not action-cards.

The deciding constraint: the action-card "Add comment…" snapshot is something the user edits — they trim items and annotate the [x]/[ ] lines before sending. That rules out any representation where the quote isn't freely-editable text.
  • Recommended keeps the quote as editable text and only decorates it — no new editor, no new source of truth, perfect send-fidelity.
  • Rich editor (B) would deliver the look but at a cost wildly out of proportion to "make a quote look like a quote."
  • Reply chip (C) is the right pattern for quoting-then-replying, but the wrong one for seed-and-annotate. Keep it in the back pocket for the reply flows only.
