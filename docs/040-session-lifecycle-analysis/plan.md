# 040 — Session Workflow & Lifecycle Analysis

## Overview

This document analyzes the complete session lifecycle in ShipIt — both what is currently implemented and what the desired/planned state looks like. Sessions are the fundamental unit of work in ShipIt: each session represents an isolated workspace with its own directory, git repo, conversation history, and preview server.

---

## 1. Session Identity

Each session has **two IDs**:

| ID | Source | Purpose |
|----|--------|---------|
| **App session ID** | `crypto.randomUUID()` generated by server | Stable identifier for workspace dirs, metadata, chat history, usage tracking. Persisted in `.vibe-sessions.json`. |
| **Agent session ID** | Claude CLI's `session_id` from `system.init` event | Used for `--resume` flag when continuing a Claude conversation. Stored as `SessionInfo.agentSessionId`. |

### SessionInfo metadata (persisted)

```typescript
{
  id: string;                                    // App session UUID
  title: string;                                 // Display name (auto-generated from first message)
  createdAt: string;                             // ISO timestamp
  lastUsedAt: string;                            // ISO timestamp (updated on every access)
  workspaceDir?: string;                         // /workspace/sessions/{uuid}/
  agentSessionId?: string;                       // Claude CLI session ID (for --resume)
  remoteUrl?: string;                            // Cached git origin URL
  archived?: boolean;                            // Soft-delete flag
  branch?: string;                               // Branch name (worktree sessions)
  sessionType?: "standalone" | "worktree";       // Type indicator
}
```

## 2. Implemented Session Lifecycle

### 2.1 Creation Flows

There are **four** ways a session gets created:

#### Flow A: Blank session via `send_message` (no sessionId)
1. User types a message on the home screen with a repo selected (or no repo)
2. Client sends `send_message` without `sessionId`
3. Server calls `createSessionDir(title)`:
   - Generates UUID → `appSessionId`
   - Creates `/workspace/sessions/{appSessionId}/`
   - Initializes empty git repo (`git init`)
   - Applies stored git identity (name/email)
   - Configures GitHub credentials if authenticated
4. Tracks in `SessionManager`
5. Initializes `ThreadManager` (creates default "main" thread)
6. Sets per-connection state: `activeAppSessionId`, `activeSessionDir`
7. Starts file watcher on the new directory
8. Spawns Claude CLI process
9. On Claude's `system.init` event → stores `agentSessionId`
10. Broadcasts `session_started` to client
11. Client navigates to `/session/{id}`

#### Flow B: Repo-backed session via `home_send_with_repo`
1. User selects a repo on the home screen and sends a message
2. Client sends `home_send_with_repo { repoUrl, text }`
3. Server computes shared repo dir: `/workspace/repos/{sha256(repoUrl)[:16]}/`
4. If shared dir doesn't exist → `git clone`; if exists → `git pull`
5. Creates worktree from shared clone: `git worktree add /workspace/sessions/{uuid} -b {branchPrefix}`
6. Session marked as `sessionType: "worktree"` with `branch` and `remoteUrl`
7. Non-blocking: spawns separate Claude CLI to generate session title + branch slug
8. On success: renames branch to `{prefix}-{slug}`, updates session title
9. Proceeds like Flow A from step 5 onward

#### Flow C: Template-based session via `home_create_repo_with_template`
1. User clicks "+ New repository" → picks template → enters repo name
2. Server creates GitHub repo, creates session, applies template files, commits, pushes to main
3. Returns `home_repo_ready { success, repoUrl, sessionId }`
4. Client pre-selects the new repo URL, user then sends a message (Flow B)

#### Flow D: Fork session via `fork_session`
1. User clicks "Fork" on an active session
2. Client sends `fork_session { branchName }`
3. Server creates worktree from shared repo (if repo-backed) or from session's own `.git` dir (standalone)
4. New session tracked with `sessionType: "worktree"`, `branch`, `remoteUrl`
5. Initializes ThreadManager for new session
6. Broadcasts `session_forked` + updated session list
7. Does **not** auto-activate the new session — user must click to switch

### 2.2 Active Use

Once a session is active, the per-connection state tracks:

```
activeAppSessionId  →  the session UUID
activeSessionDir    →  /workspace/sessions/{uuid}/
```

All operations use these to scope work:
- **File operations** (`get_file_tree`, `read_file`, etc.) → read from `activeSessionDir`
- **Git operations** (`get_git_log`, `git_push`, etc.) → `GitManager(activeSessionDir)`
- **Preview server** → serves from `activeSessionDir`
- **File watcher** → watches `activeSessionDir`
- **Claude CLI** → `--cwd activeSessionDir`
- **Chat history** → appended to `/workspace/.vibe-chat-history/{sessionId}.json`

#### Message flow during active use
1. Client sends `send_message { text, sessionId }`
2. If `sessionId` matches active session → resume with `--resume {agentSessionId}`
3. Claude streams events (text, tool_use, tool_result)
4. On `result` event → auto-commit via `GitManager.autoCommit()`
5. If remote configured → debounced auto-push (5s trailing delay)
6. If preview not running → `previewManager.start(activeDir)`
7. Port scan for dev servers

#### Thread operations within a session
- **Create checkpoint** → captures `{ messageIndex, commitHash }` on active thread
- **Fork thread** → snapshot thread data, `git reset --hard` to checkpoint commit, restore thread data, create new thread
- **Switch thread** → rollback git to target thread's checkpoint, load corresponding chat history

### 2.3 Session Switching

Triggered by clicking a session in the sidebar (`get_chat_history { sessionId }`):

1. Server calls `activateSession(sessionId)`:
   - Looks up session metadata from `SessionManager`
   - If different from current session:
     - Stops preview server
     - Kills processes on detected ports (safety-guarded to `/workspace/sessions/` only)
     - Clears detected ports
     - Broadcasts `preview_status { running: false }`
     - Clears terminal log buffer, broadcasts `clear_logs`
     - Stops file watcher, restarts on new directory
     - Starts preview for new directory
     - Runs port scan
     - Checks/applies git identity
2. Sets `activeAppSessionId` and `activeSessionDir`
3. Loads and broadcasts `chat_history` for the session
4. Client resets ALL UI state (messages, files, git commits, threads, usage, etc.)
5. Client sends `get_file_tree` + `get_git_log` to refresh workspace data

**Edge case: worktree directory missing** — if the worktree session's directory no longer exists (e.g., manually deleted), the server sends chat history but also an error message, and does **not** activate the session.

### 2.4 Creating a New Session (clearing active)

Triggered by the "+" button in the sidebar (`new_session`):

1. Server clears `activeAppSessionId` and `activeSessionDir`
2. Clears message queue (prevents cross-session bleed)
3. Stops preview server
4. Kills processes on detected ports
5. Clears log buffer, broadcasts `clear_logs`
6. Broadcasts updated session list
7. Client clears all state, shows template/home screen

No session directory is created yet — that happens on the next `send_message` or `home_send_with_repo`.

### 2.5 Archiving (Soft Delete)

Triggered by archive icon in sidebar (`archive_session { sessionId }`):

1. **Worktree cleanup** (if `sessionType === "worktree"`):
   - Finds parent repo dir via `remoteUrl` or `.git` file parse
   - `git worktree remove --force {workspaceDir}`
   - `git branch -D {branch}`
2. Sets `archived: true` in SessionManager
3. If archiving the active session → clears per-connection state
4. **Shared repo cleanup**: if no non-archived sessions remain for this `remoteUrl`:
   - Deletes entire `/workspace/repos/{hash}` directory
5. Broadcasts updated session list

**What is preserved**: workspace directory (for standalone), chat history files, thread data, usage data. Only the worktree checkout and branch are removed for worktree sessions.

### 2.6 Full Reset

Triggered by "Reset Everything" in settings (`full_reset`):

1. Stops Claude process, preview server, file watcher
2. Deletes ALL persistent data: sessions, chat history, threads, usage, system prompt, git identity, GitHub token, deployment configs
3. Does NOT clear Claude CLI OAuth (lives outside `/workspace/`)
4. Client clears localStorage, reloads page

### 2.7 Rename

Triggered by inline edit in sidebar (`rename_session { sessionId, title }`):

1. Validates title is non-empty
2. Updates SessionManager
3. Broadcasts `session_renamed`

## 3. Session State Matrix

### Per-connection (in-memory, lost on disconnect)

| State | Set when | Cleared when |
|-------|----------|--------------|
| `activeAppSessionId` | Session created or switched to | `new_session`, `archive_session` (if active), disconnect |
| `activeSessionDir` | Session created or switched to | Same as above |
| `claudeProcess` | Claude spawned | Process exits, session switch |
| File watcher | Session activated | Session switch, `new_session` |
| Preview server | Session activated or post-turn | Session switch, `new_session` |
| Detected ports | Port scan | Session switch, `new_session` |
| Message queue | Messages queued while Claude busy | Queue processed, session switch, `new_session` |
| Log buffer | Claude/server output | Session switch, `new_session` |

### Per-session (persisted to disk)

| Data | Location | Created | Deleted |
|------|----------|---------|---------|
| Session metadata | `.vibe-sessions.json` | `createSessionDir()` | `archive_session` (soft), `full_reset` |
| Workspace directory | `/workspace/sessions/{uuid}/` | `createSessionDir()` | `archive_session` (worktree only), `full_reset` |
| Chat history | `.vibe-chat-history/{id}.json` | First message append | `full_reset` |
| Thread data | `.vibe-threads/{id}.json` | `ThreadManager.init()` | `full_reset` |
| Git repo | Inside workspace dir | `git init` or `git worktree add` | With workspace dir |
| Usage data | `.shipit-usage.json` | First usage record | `full_reset` |

### Client-side (reset on session switch)

All of these are cleared when switching sessions or creating new:

```
messages, fileTree, gitCommits, viewingFile, viewingFileContent,
threads, activeThreadId, currentSessionUsage, modelInfo,
contextTokens, turnTokens, preview, selectedPort, previewErrors,
autoFixEnabled, autoFixRetries, activity, logEntries, unreadLogCount,
shellStarted, terminalMode, queuedMessages
```

**Not reset** (user preferences): theme, sidebar collapsed state, permission mode, agent preference, panel split ratio.

## 4. Directory Structure

```
/workspace/
  .vibe-sessions.json              ← SessionManager (all session metadata)
  .vibe-chat-history/
    {sessionId}.json               ← chat messages per session
    {sessionId}__{threadId}.json   ← chat messages per forked thread
  .vibe-threads/
    {sessionId}.json               ← thread/checkpoint data per session
  .shipit-usage.json               ← per-turn cost/duration tracking
  .shipit/
    system-prompt.md               ← custom system prompt
    git-identity.json              ← global git name/email
  .github-token                    ← GitHub OAuth token
  .shipit-deploy/                  ← deployment credentials/history
  sessions/
    {uuid-A}/                      ← standalone session workspace
      .git/                        ← full git repo
      [project files]
    {uuid-B}/                      ← worktree session workspace
      .git                         ← worktree link file (not a directory)
      [project files]
  repos/
    {sha256-hash}/                 ← shared repo clone (one per unique URL)
      .git/
        worktrees/
          {uuid-B}/                ← worktree metadata
      [main working tree files]
```

### Architectural Observations & Implicit Gaps

#### 1. No workspace cleanup on archive (standalone sessions)
When archiving a **standalone** session (not a worktree), the workspace directory at `/workspace/sessions/{uuid}/` is preserved. Only worktree sessions get their checkout removed. Over time, archived standalone sessions accumulate disk usage. There is no garbage collection mechanism.

**Current behavior**: `archived: true` flag hides the session from the sidebar, but files remain.
**Desired behavior (implicit)**: Either delete the workspace on archive, or provide a periodic cleanup mechanism. The `full_reset` is the only way to reclaim space.

#### 2. No chat history cleanup on archive
Chat history files (`/workspace/.vibe-chat-history/{sessionId}.json`) and thread data files are preserved even after archive. Again, only `full_reset` removes them.

#### 3. Session data scattered across multiple files
A single session's state is spread across:
- `.vibe-sessions.json` (metadata)
- `.vibe-chat-history/{id}.json` (messages)
- `.vibe-threads/{id}.json` (threads/checkpoints)
- `.shipit-usage.json` (usage — shared file, not per-session)
- `/workspace/sessions/{id}/` (workspace files)

This makes it impossible to "export" or "move" a session atomically. The session sharing feature (023, paused) would need to gather from all these sources.

#### 4. Single-connection model
Per-connection state (`activeAppSessionId`, `activeSessionDir`) means only one browser tab can be active at a time. If a second tab connects, both track independent state but share the same server-side managers. This can cause:
- Conflicting `activateSession()` calls
- Claude processes from one tab's session interfering with another
- `full_reset` from one tab surprising the other

**Mitigation**: The product targets single-user single-tab usage. Multi-tab is unsupported but not prevented.

#### 5. No session migration path
Sessions created before the worktree system (`sessionType` undefined, treated as standalone) have no migration path to become worktree sessions. If a user wants to connect an old standalone session to a shared repo, they must archive it and create a new worktree session.

#### 6. Claude CLI session continuity on reconnect
When the WebSocket disconnects and reconnects (browser refresh, network blip), the client re-sends `get_chat_history` to restore UI state. The `agentSessionId` is persisted, so the next `send_message` can `--resume` the Claude CLI conversation. However:
- Any in-flight Claude response is lost (the process dies with the connection)
- The message queue is cleared on disconnect
- No "reconnect to running Claude process" — it's a fresh spawn

#### 7. Thread data in git working tree
Thread data files live at `/workspace/.vibe-threads/{sessionId}.json`, which is **outside** the session workspace dir but still under `/workspace/`. The snapshot-before-rollback pattern protects against `git reset --hard` in the session dir. However, thread data is shared across all sessions (each session gets its own file in the same directory). This is fine for isolation but means threads are tied to the container, not portable.

#### 8. Preview server lifecycle is tied to session activation
The preview server starts when a session is activated and stops on switch. There is no "keep preview running in background" for quick back-and-forth between sessions. Each switch incurs a preview restart cost (potentially slow for projects with long install/build steps).

## 7. WebSocket Message Summary (Session-Related)

### Client → Server

| Message | Purpose | Creates Session? |
|---------|---------|:---:|
| `send_message` (no sessionId) | Start new blank session | Yes |
| `send_message` (with sessionId) | Resume existing session | No |
| `home_send_with_repo` | Start repo-backed worktree session | Yes |
| `home_create_repo_with_template` | Create GitHub repo + session | Yes |
| `fork_session` | Fork active session as worktree | Yes |
| `get_chat_history` | Switch to / load a session | No (activates) |
| `new_session` | Clear active session state | No |
| `archive_session` | Soft-delete a session | No |
| `rename_session` | Change session title | No |
| `list_sessions` | Fetch all non-archived sessions | No |
| `full_reset` | Wipe everything | No (destroys all) |

### Server → Client

| Message | Purpose |
|---------|---------|
| `session_started` | New session created (includes session metadata) |
| `session_list` | Full list of non-archived sessions |
| `session_renamed` | Session title updated |
| `chat_history` | Persisted messages for a session |
| `session_forked` | Worktree session created from fork |
| `worktree_list` | All worktrees sharing same repo |
| `full_reset_complete` | Container wiped successfully |

## 8. Summary

The session system is well-implemented for its current scope. The core lifecycle (create → use → switch → archive) works reliably with proper isolation. Key strengths:

- **Strong isolation**: Each session has its own workspace, git repo, and file watcher
- **Worktree efficiency**: Repo-backed sessions share a single clone, saving disk and clone time
- **Clean state management**: Session switches properly tear down old state (preview, ports, logs, file watcher)
- **Thread branching**: Checkpoints + thread forks allow conversation exploration without losing work

Key areas that remain unaddressed:
- **Disk cleanup**: Archived standalone sessions accumulate indefinitely
- **Session export**: Sharing/exporting sessions is designed (doc 023) but paused
- **Multi-tab**: No support for concurrent browser sessions
- **Session portability**: No import/export or migration between standalone ↔ worktree
