# Installable PWA

Make ShipIt installable on mobile so it runs **standalone** ŌĆö full-screen, no
Chrome address bar ŌĆö as an alternative to the Android WebView wrapper
(`android/`), which carries a lot of quirks. Hard requirement: the installed app
must **always boot the latest code**. ShipIt is a live, server-driven IDE; a
stale cached shell or JS bundle is actively harmful (an old client talking to a
newer orchestrator), so this PWA deliberately introduces **no caching**.

## Why a service worker at all

The manifest, icons, theme-color and Apple meta tags were already present, but a
browser will not offer "Add to Home Screen" / installation as a standalone app
without a registered **service worker that has a `fetch` handler**. So the
worker exists purely to satisfy installability ŌĆö not to cache.

## How "always latest" is preserved

Three layers, none of which add a cache:

1. **Cache-free service worker** (`src/client/public/service-worker.js`).
   Precaches nothing. Its `fetch` listener never calls `respondWith`, so every
   request goes straight to the network exactly as if no worker were installed.
   On `activate` it deletes any Cache Storage left by a previous worker version
   (so upgrading from a hypothetical caching worker can't strand a user on old
   assets), and `skipWaiting` + `clients.claim` take control immediately.

2. **`no-store` on the shell + worker** (`serveStaticClient` in
   `src/server/orchestrator/app-assembly.ts`). `@fastify/static`'s built-in
   `cacheControl` is disabled and headers are set explicitly: `*.html` and
   `service-worker.js` get `no-store, must-revalidate`; everything else
   (Vite's content-hashed `assets/*`) keeps the prior `max-age=0,
   must-revalidate` always-revalidate behavior. No new caching is introduced.

3. **Existing build-id reload** (`useServerEvents` + `client-build.ts`,
   unchanged). When the orchestrator advertises a new `buildId` over SSE, the
   open client calls `window.location.reload()`. Because the shell is
   `no-store` and the worker caches nothing, that reload always fetches the new
   hashed bundle. The worker is registered with `updateViaCache: "none"` and an
   explicit `registration.update()` on load so the worker script itself is also
   re-checked against the network every load.

## Key files

- `src/client/public/service-worker.js` ŌĆö the cache-free worker.
- `src/client/utils/register-service-worker.ts` ŌĆö registration helper (root
  scope, `updateViaCache: "none"`, failures swallowed so a bad registration
  never breaks boot). Called once from `src/client/main.tsx`.
- `src/server/orchestrator/app-assembly.ts` ŌĆö `serveStaticClient` sets the
  Cache-Control headers.
- `src/client/index.html` ŌĆö already carried the manifest link, theme-color and
  Apple meta tags; `apple-touch-icon` now points at the PNG.
- `src/client/public/manifest.webmanifest` ŌĆö adds PNG icons (192/512,
  `any maskable`) alongside the SVG.
- `src/client/public/{icon-192,icon-512,apple-touch-icon}.png` ŌĆö full-bleed
  raster icons. iOS ignores SVG `apple-touch-icon` and renders transparent
  corners as black, so the raster icons are full-bleed (gradient to the edge)
  and the OS applies its own corner mask. Generated by rendering `favicon.svg`
  in a browser at each size (no new build dependency).

## Tests

- `src/server/orchestrator/app-assembly.test.ts` ŌĆö `index.html`, the SPA
  fallback and `service-worker.js` are served `no-store`; hashed assets are not.
- `src/client/utils/register-service-worker.test.ts` ŌĆö registers at root scope
  with cache bypass; no-op when unsupported; swallows registration failures.

## Notes / future

- This is intended to reduce reliance on the Android WebView wrapper
  (`android/`, docs/116). The wrapper still exists; this is a parallel path.
- If offline support is ever wanted it would be a deliberate, separate decision
  ŌĆö it conflicts with the always-latest requirement and is explicitly out of
  scope here.
